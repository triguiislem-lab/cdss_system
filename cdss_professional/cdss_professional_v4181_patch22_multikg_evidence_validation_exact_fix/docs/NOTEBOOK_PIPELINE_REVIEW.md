## Scope

This document reviews the notebooks at the project root and the way their logic is integrated into the automatic prescription system:

- `notebook38f27ecf91 (2).ipynb`: knowledge graph construction and graph comparison notebook.

## Model Execution Verification

The project is advanced and model-capable, but the active `/prescriptions/draft` path should be separated from optional/model-ready components.

Active settings source:

- The API imports `get_settings()` from `libs/config/settings.py` through `libs/config/__init__.py`.
- The root-level `settings.py` also exists and is kept aligned for documentation/legacy use, but the API/container path does not import this file.
- Therefore, the active defaults are the ones in `libs/config/settings.py` and `.env`/`.env.tunisia_runtime`.

Model-capable code that can load real models:

- FAISS vector retrieval can load `SentenceTransformer(VECTOR_EMBEDDING_MODEL)` when `VECTOR_BACKEND=faiss`.
- Translation can load `facebook/nllb-200-distilled-1.3B` through Hugging Face `transformers.pipeline`.
- Deduplication can load `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`.
- Generation can call an OpenAI-compatible endpoint or `llama_cpp` if configured.

What now runs by default in `/prescriptions/draft`:

- Clinical understanding: NLLB translation is injected into `TranslationStep`; if the model/dependencies are unavailable, phrase-map normalization is used as fallback and the status is recorded in `extracted_context.translation_model_status`.
- Retrieval: `semantic_jsonl` loads the final evidence release, lexical-prefilters candidates, then uses `SentenceTransformer(BAAI/bge-m3)` to semantically rerank when available.
- Reranking: `EvidenceRanker` now attempts `CrossEncoder(BAAI/bge-reranker-v2-m3)`, then `cross-encoder/ms-marco-MiniLM-L6-v2`, then heuristic fallback.
- Deduplication: `RetrievalService` receives `DeduplicationService` and attempts `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` deduplication.
- Generation: OpenAI-compatible Qwen3-32B endpoint is tried first; successful model output is marked with `llm_model_used=true`.
- Safety: deterministic clinical guardrail rules.
- Localization: deterministic AMM matching and product ranking.

What still requires external runtime availability:

- `Qwen3-32B` requires a running OpenAI-compatible local server at `GENERATION_BASE_URL`, or another configured endpoint.
- Hugging Face models require installed dependencies and locally available/downloadable weights.
- If a model cannot load, the runtime falls back and records model-used/status metadata rather than pretending the model ran.

Conclusion: the default pipeline is now model-preferred for translation, semantic retrieval, reranking, deduplication, and generation. Deterministic logic remains as a safety fallback and for clinical guardrails.

## Prescription Generation Pipeline

Runtime entrypoint:

- API: `apps/api/routers/prescriptions.py`
- Container wiring: `apps/api/container.py`
- Main orchestrator: `services/orchestration/pipeline.py`

### Step 1: Receive Request

The API receives a `DraftPrescriptionCommand` containing:

- `request_id`
- `patient`
- `consultation`

The pipeline creates a trace ID and resets stage tracing.

### Step 2: Clinical Understanding

Module:

- `services/clinical_understanding/service.py`

Logic:

1. Normalize consultation language/text through `TranslationStep`.
2. Build runtime text from consultation transcript and doctor notes.
3. Parse symptoms, suspected conditions, missing information, disease tags, vulnerable flags, and extracted context.
4. Extract patient risk flags from profile and parsed context.
5. Build a `PatientSnapshot`.
6. Route the case through `ProductionRouter`.

Possible route recommendations:

- `prescription`
- `review`
- `emergency`
- `non_pharma`

This route controls whether automatic medication drafting/localization is allowed.

### Step 3: Retrieval Query Planning

Module:

- `services/retrieval/query_builder.py`

Logic:

1. Collects normalized symptoms, suspected conditions, chronic conditions, current medications, allergies, and risk tokens.
2. Adds patient risk tokens:
   - pregnancy
   - breastfeeding
   - renal impairment
   - hepatic impairment
   - drug allergy
   - elderly
   - pediatric
3. Maps known diseases/syndromes to runtime aliases:
   - influenza/viral syndrome/URTI -> `grippe`
   - cardiac emergency/STEMI -> `stemi`
   - myopia/refractive error -> `myopie`
   - migraine/headache -> `migraine`
   - diabetes -> `diabete`
   - asthma -> `asthme`
   - dental abscess -> `abces_dentaire`
4. Builds three retrieval queries:
   - vector query with prescription hints
   - KG query with disease/condition terms
   - local formulary query with disease/condition terms
5. Adds filters:
   - vector: language, route, disease when available
   - KG: route, disease when available
   - local formulary: market `TN`

### Step 4: Hybrid Retrieval

Module:

- `services/retrieval/hybrid_retriever.py`

The retriever combines:

1. Vector/text evidence from `VectorRetriever`.
2. Knowledge graph facts from `KGRetriever`.
3. Tunisian local products from `LocalFormularyRetriever`.

Default limits:

- 5 vector chunks
- 5 KG facts
- 5 local products

### Step 5: Runtime Data Retrieval

Vector/text retrieval:

- Default backend: semantic JSONL scoring.
- File: `final_data_release/final_evidence_sections_runtime.jsonl`
- Scoring uses lexical prefiltering, then BGE-M3 embeddings when available. If the embedding model cannot load, lexical fallback is returned with model-status metadata.

KG retrieval:

- Default backend: CSV.
- File: `data/runtime/tn_master_kg_edges.csv`
- Scoring uses lexical overlap plus predicate priority.

Local formulary retrieval:

- Default backend: CSV.
- File: `final_data_release/final_medicines.csv`
- Records become `LocalProductEvidence` with product name, active ingredient, strength, form, market, and metadata.

### Step 6: Evidence Reranking

Module:

- `services/retrieval/evidence_ranker.py`

Current logic is heuristic:

1. Assign source bucket:
   - structured drug knowledge
   - guideline
   - safety rule
   - textbook
   - exam QA
   - general
2. Apply bucket weights:
   - structured drug knowledge: `+0.22`
   - guideline: `+0.16`
   - safety rule: `+0.14`
   - textbook: `+0.05`
   - general: `+0.0`
   - exam QA: `-0.14`
3. Add exact term hit boost:
   - up to `+0.07`
   - `0.015` per exact hit
4. Add patient-context boost:
   - pregnancy evidence: up to `+0.04`
   - renal evidence: up to `+0.04`
   - elderly evidence: up to `+0.02`
   - pediatric evidence: up to `+0.02`
   - total capped at `+0.08`

The configured reranking models are now actively attempted through `sentence_transformers.CrossEncoder`. If model loading fails, the ranker falls back to the previous heuristic and records `reranker_model_used=False` in metadata for evidence chunks/products.

### Step 7: Evidence Fusion

Module:

- `services/retrieval/evidence_fuser.py`

The fuser creates an `EvidenceBundle` containing:

- ranked vector chunks
- ranked graph facts
- ranked local products
- retrieval plan
- merged summary

The summary records primary terms, patient context, top text evidence, top graph fact, top local product, and active filters.

### Step 8: Therapy Strategy Detection

Module:

- `services/generation/therapy_strategy.py`

The system decides how to use Tunisia evidence:

- `non_pharma`: optical/non-pharmacologic case; no medication draft.
- `emergency`: emergency/red flag; medication draft suppressed.
- `symptomatic`: symptomatic case; Tunisia data used mainly for localization, not regimen authority.
- `review`: safety/adjustment context; clinician review.
- `disease_directed`: disease-directed case; Tunisia evidence can support regimen authority.

Examples:

- Myopia/lunettes/diopter terms -> `non_pharma`
- Chest pain/STEMI/anaphylaxis/stroke/severe dyspnea -> `emergency`
- Flu/viral/fever/cough/sore throat/pain/headache -> `symptomatic`
- Pregnancy/renal/hepatic/review/deprescribing -> `review`

### Step 9: Candidate Selection

Module:

- `services/generation/candidate_selector.py`

Candidate logic:

1. Scores local products by ingredient, indication match, and local availability.
2. Scores KG treatment facts when predicates indicate treatment or recommendation.
3. Adds vector support when retrieved text mentions an ingredient.
4. Adds notebook-inspired symptomatic defaults:
   - paracetamol for fever, headache, pain, sore throat, viral syndrome
   - ibuprofen for pain/headache only when NSAID risk is absent
   - salbutamol for asthma/bronchospasm/wheeze context
5. Applies penalties:
   - NSAIDs get `+1.25` penalty if pregnancy, renal impairment, NSAID allergy, anticoagulants, peptic ulcer, anticoagulation, or asthma risk exists.
   - Paracetamol gets `+0.10` penalty in hepatic impairment.
6. Keeps candidates with adjusted score above `0.18`.
7. Returns at most 2 candidates.

### Step 10: Draft Generation

Module:

- `services/generation/llm_router.py`

Default backend:

- `openai_compatible`

The heuristic generator renders structured compact text with fields:

- `problem_summary`
- `triage`
- `confidence`
- `medication`
- `support`
- `non_drug`
- `monitoring`
- `question`
- `note`

Built-in regimen templates currently include:

- paracetamol:
  - default: `500 mg`, every 8 hours, 3 days, oral
  - hepatic impairment: `500 mg`, every 12 hours, 2 days
- ibuprofen:
  - `400 mg`, every 8 hours, 3 days, oral
- salbutamol:
  - `100 mcg/dose`, as needed, 5 days, inhalation
- dextromethorphan:
  - `15 mg`, every 8 hours, 3 days, oral, only for cough context

If an external LLM backend is configured, the router sends an OpenAI-compatible chat payload with:

- `model`
- system prompt
- user prompt
- `temperature`
- `max_tokens`

If the external call fails or is not configured, it falls back to the heuristic path.

### Step 11: Output Parsing And Normalization

Module:

- `services/generation/output_parser.py`

The parser supports:

- compact line-based output
- notebook-style markdown output with sections like `## DIAGNOSIS` and `## PROPOSED PRESCRIPTION`

Normalization logic:

1. Canonicalizes active ingredients.
2. Removes unknown/empty medications.
3. Marks incomplete dose/frequency/duration as unresolved.
4. Maps triage labels to contract values.
5. Suppresses medications for emergency and non-pharma routes.
6. Forces clinician review if dosing details are incomplete.

### Step 12: Contract Enforcement

Module:

- `services/orchestration/pipeline.py`

After generation, the pipeline enforces route-level behavior:

- `emergency`: medication table suppressed, emergency referral added.
- `non_pharma`: medication draft suppressed.
- `review`: clinician review required.
- unresolved dosing: clinician review required.

### Step 13: Deterministic Safety Validation

Module:

- `services/safety/service.py`

Safety checks run after generation:

- allergy rules
- drug-drug interaction engine
- contraindication rules
- pregnancy rules
- renal rules
- hepatic rules
- dose guardrails
- escalation rules

Findings are deduplicated and sorted with critical findings first.

Safety knowledge handling:

- Runtime safety source is based on Tunisian runtime medicine evidence when available.
- A small `examples/demo_fixtures/safety_rules_stub.json` fallback remains for defensive classic high-risk guardrails.
- The fallback is not considered authoritative clinical evidence by itself.

### Step 14: Blocking Decision

The pipeline blocks if:

- safety has a blocking issue, or
- route recommendation is `review`, `emergency`, or `non_pharma`.

If blocked and `localize_blocked_plans=False`, localization is skipped.

Default:

- `localize_blocked_plans=False`

This is important: unsafe or incomplete cases do not proceed to Tunisian product localization by default.

### Step 15: Tunisian Localization

Module:

- `services/localization/tunisia_localizer.py`

Only runs when not blocked, unless configured otherwise.

Localization steps:

1. Rank local Tunisian products.
2. Map generic medication draft to AMM candidates.
3. Choose best strength/form match.
4. Add reimbursement/VEI note.
5. Return localized medications sorted by confidence.

AMM scoring in `services/localization/amm_mapper.py`:

- exact DCI match: `80`
- same ingredient set: `76`
- query set subset of row set: `68`
- row set subset of query set: `58`
- substring DCI match: `52`
- partial ingredient intersection: `40 + 4 * intersection size`
- route/form match: `+15`
- route/form mismatch: `-18`
- dose token match: `+16`
- partial mg token match: `+8`
- dose mismatch: `-3`
- indication bonus: up to `+6`
- VEIC vital bonus: `+2.5`
- VEIC essential bonus: `+1.5`
- generic bonus: `+0.5`
- candidates require raw score `>35`

### Step 16: Proposal And Audit

The pipeline returns a `PrescriptionProposal` with:

- generic draft plan
- localized Tunisian medications
- clinician review requirement
- review notes
- blocking reasons

Then it writes an audit record through:

- `services/audit/service.py`
- `services/audit/repository.py`

Default audit backend:

- file audit in `data/audit`
